# MockMate AI — Dashboard & Post-Interview Scoring Logic

**Version:** 1.0  
**Scope:** Developer guide and implementation strategy for the Dashboard metrics, polling mechanisms, and the comprehensive hybrid scoring system.

---

## 1. Dashboard Architecture & Implementation Strategy

The dashboard is the central hub for the user to review their performance and start new sessions. It requires real-time responsiveness and accurate data aggregation.

### 1.1 Layout Grid
The dashboard follows a 2-column layout (`lg:grid-cols-5`):
- **Left Column (3/5 width):** Practice Sessions list (LIFO stack - newest first).
- **Right Column (2/5 width, sticky):** Sidebar containing Quick Stats, Overall Performance, Skill Breakdown, and AI Recommendations.

### 1.2 State Management & Polling Strategy
- **Session Data Source:** The dashboard fetches the list of sessions from `GET /sessions/`.
- **Polling Loop:** A `useEffect` hook must poll the backend every 10 seconds **only if** there is at least one session in the `creating` or `live` state. Once all sessions reach `completed`, `interrupted`, or `failed`, polling stops to save network resources.
- **Optimistic UI Updates:** When a new session is created via the Create Session Modal, it should be immediately unshifted to the local state array while the background AI generation runs.

### 1.3 Sub-components
1. **Quick Stats:**
   - **Total Sessions:** Count of all session objects.
   - **Average Score:** Average of the `overall_score` across all `completed` sessions.
   - **Total Practice Time:** Sum of `duration` across all completed sessions.
2. **Overall Performance (from the MOST RECENT completed session):**
   - Displays 4 key metrics: Overall Score, Fluency, Content Quality, and Confidence.
   - Trend arrows evaluate the delta between the *latest* session and the *previous* session.
3. **Skill Breakdown:**
   - Visualized as 5 progress bars parsed from the latest session's `scores.skills` array.
4. **AI Recommendations:**
   - Render lists mapping to `ai_suggestions_behavioral` and `ai_suggestions_technical` from the latest report payload.

## 2. Interview Scoring System — Logic & Strategies

MockMate AI utilizes a **Hybrid Scoring System** (70% AI + 30% Rule-based) to ensure evaluations are both deeply intelligent regarding content, and mathematically objective regarding delivery.

### 2.1 Per-Question Base Score Calculation

Every answered question goes through a two-step evaluation *before* the transcript is spell-corrected.

`Final Question Score = (0.7 * AI_Content_Score) + (0.3 * Rule_Score)`

**A. AI Content Score (0-100)**
Generated by Gemini directly analyzing the text prompt + raw answer.
- **Criteria:** Relevance to the prompt, technical depth, behavioral accuracy, and structural coherence.

**B. Rule-Based Score (0-100)**
Calculated deterministically by the Python Backend.
- **Filler Word Penalty:** Count of "um", "uh", "like", "you know". (e.g., `-2` points per filler word, floored at 0).
- **Pace Penalty:** Ideal speaking rate is 2.5–4 words per second. Deviation outside this band incurs a percentage reduction.
- **Silence/Pause Penalty:** Tracking gaps >1.5s. Extensive mid-answer silence drops the score.

### 2.2 System Multipliers (Applied to Per-Question Scores)

1. **Difficulty Multiplier:** 
   - Easy: 1.0x baseline.
   - Moderate: Scores are scaled or graded against a harder baseline.
   - Hard: Requires exceptional depth to achieve >80% on AI content.
2. **Module Weighting:**
   - Module 1 (Resume-based, 35% of time): Standard weighting.
   - Module 2 & 3 (JD-based core skills, 65% of time): Given 1.5x weight in the final aggregation as this determines actual job competency.

### 2.3 Aggregated Dashboard Metrics

Once the per-question scores are calculated, the backend aggregates them into the high-level metrics displayed on the dashboard.

1. **Fluency (0-100):**
   - Derived mathematically from consistent speaking pace, minimized silence gaps, and low filler-word frequency.
2. **Confidence (0-100):**
   - Derived from AI semantic analysis (did they sound assertive and clear?) combined with hesitation metadata.
3. **Content Quality (0-100):**
   - The direct average of the `AI_Content_Score` across all answered questions. 
4. **Overall Final Score (0-100):**
   - The weighted average of all post-multiplier Question Scores.

### 2.4 Skill Dimensions Distribution

The total score is also fragmented into 5 specific skills. The weight of questions applied to each skill changes based on the **JD Classification** (determined automatically during session creation).

| Skill | Technical JD Weight | Managerial JD Weight |
|-------|---------------------|----------------------|
| Technological Knowledge | 40% | 20% |
| Communication | 15% | 20% |
| Problem Solving | 20% | 15% |
| Leadership | 10% | 25% |
| Cultural Fit | 15% | 20% |

*If a role is classified as Technical, scoring high on technical questions heavily drives the final score. If Managerial, the weight shifts toward communication and leadership answers.*

## 3. Recommended Developer Strategies

1. **Decouple Evaluation from API Response:**
   - Post-interview evaluation takes time. The frontend MUST NOT block waiting for the report. The backend should return `202 Accepted` and run the evaluation as a FastAPI background task. The dashboard correctly polls for completion.
2. **Preserve Raw Audio Metrics:**
   - Do not pass the spell-corrected transcript to the rule-based scorer. Filler words and pauses must be analyzed on the raw STT stream to ensure Fluency and Confidence scores are accurate.
3. **Robust Fallbacks:**
   - If a user answers 0 questions (e.g., immediate abort), the dashboard should gracefully render an `N/A` or bypass the scoring metrics rather than throwing a divide-by-zero error on the UI.

## 4. API Costs & Model Deep Analysis

Selecting the right Gemini model for each backend operation is critical for balancing system latency, accuracy, and operational cost.

### 4.1 Available Models & Cost Comparison

*Note: Pricing is approximate based on Google Gen AI tiering per 1 Million (1M) Tokens.*

| Model Selection | Cost: Input / 1M Tokens | Cost: Output / 1M Tokens | Primary Strength |
|-----------------|-------------------------|--------------------------|------------------|
| **Gemini 1.5 Flash** | ~$0.075 | ~$0.30 | **Ultra-low latency**, cost-efficiency, high-volume speed |
| **Gemini 1.5 Pro** | ~$3.50 | ~$10.50 | **Deep reasoning**, strict JSON adherence, nuanced evaluation |
| **Gemini 2.0 Flash** | (Varies / Beta) | (Varies / Beta) | Next-gen tradeoff of speed & intelligence, but historically restricted for new endpoints (`404` errors) |

### 4.2 Latency vs. Output Efficiency

1. **Gemini 1.5 Flash (The Speed Engine)**
   - **Latency:** Outstanding (< 500ms time-to-first-byte in streaming).
   - **Output Efficiency:** Great for unstructured-to-structured simple mapping (like Resume parsing boundaries) and rapid generative tasks (like generating follow-up questions live).
   - **Drawback:** Can sometimes struggle with highly complex multi-stage reasoning or deep abstract evaluation constraints.

2. **Gemini 1.5 Pro (The Reasoning Engine)**
   - **Latency:** Noticeably slower (often 1.5s - 3s response time for moderate generation).
   - **Output Efficiency:** Unmatched for strict JSON schema enforcement, nuanced candidate scoring, and extracting specific pedagogical coaching advice.
   - **Drawback:** ~45x more expensive than Flash, making it unviable for real-time WebSocket conversational audio streaming.

### 4.3 Recommended Model Mapping for MockMate Architecture

To achieve the best system performance at the lowest API cost, the `gemini_service.py` functions should be mapped distinctly rather than using a single global model fallback:

- **Resume Parsing (`parse_resume`)**: `gemini-1.5-flash`
  - *Why:* Resume formatting is standard. Flash handles JSON schema mapping fast and cheap.
- **JD Quality & Analysis (`analyze_jd_quality`, `classify_jd_type`)**: `gemini-1.5-flash`
  - *Why:* Simple text classification task. Does not require deep reasoning.
- **Question Generation (`generate_resume_questions`, `generate_jd_questions`)**: `gemini-1.5-pro`
  - *Why:* We need highly contextual, intelligent questions that align with the specific JD. Pro provides better quality, and this happens *before* the interview starts, so the user doesn't feel the latency.
- **Real-Time Follow-ups (`generate_followup_question`)**: `gemini-1.5-flash`
  - *Why:* The user is actively waiting mid-interview. Speed is the absolute highest priority to prevent awkward silence.
- **Post-Interview AI Scoring & Evaluation (`evaluate_answer`, `generate_question_feedback`)**: `gemini-1.5-pro`
  - *Why:* This determines the user's score and coaching. Accuracy and deep reasoning are non-negotiable. This runs in the background while the user views the "Evaluating..." spinner.
- **Real-Time Speech (STT/TTS WebSocket)**: **Gemini Live API** (powered by Flash backend)
  - *Why:* Specifically optimized for real-time Voice-to-Voice latency bounds without manual token chunking overhead.
